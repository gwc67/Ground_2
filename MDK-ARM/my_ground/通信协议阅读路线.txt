零、先建立两个心理模型（不懂这两个，代码白读）

【模型 1：串口在单片机里怎么工作 —— 中断 vs DMA】

  你用 PC 的串口，是"主动调用 read()，数据就来了"。
  但单片机里没人"主动调"，数据是硬件随时、不定时、一个字节一个字节地到。
  必须有一种机制在"数据到了"的瞬间通知 CPU 去收。两种做法：

  ┌─────────────┬─────────────────────────────────────────────┐
  │ 中断(IT)     │ 每来 1 个字节，硬件打断 CPU 一次                │
  │              │ 优点：好懂、几乎不需额外硬件                    │
  │              │ 缺点：数据快了 CPU 被打断太频繁（波特率高时累）  │
  ├─────────────┼─────────────────────────────────────────────┤
  │ DMA          │ 专门的硬件模块，自己把字节搬进一块内存          │
  │              │ 不打断 CPU，搬到一段结束（或空闲）才通知一次     │
  │              │ 优点：CPU 不被打扰，高速通信首选               │
  │              │ 缺点：要会配 DMA、要懂"搬完一段"怎么算          │
  └─────────────┴─────────────────────────────────────────────┘

  这个项目三条 UART 各自不同：
    USART1 (ground)：DMA 收 + DMA 发  ← 最常用，飞控通信
    USART2 (screen)：DMA 收 + DMA 发  ← 屏幕通信
    USART3         ：IT 收  + IT 发    ← 慢速/简单，电池电压等

  关键：DMA 接收到的字节，是先堆在一块"接收缓冲区"里，等任务去取。
  任务怎么知道"有多少新数据"？两种：本工程用 HAL 的 "ReceiveToIdle"
  (收到一个空闲间隔=一帧结束，触发回调，告诉你这次收到多少字节)。


【模型 2：为什么需要 RingBuffer（环形缓冲区）】

  收数据（生产）和 处理数据（消费）速度不一样、时机不一样：
    - 收数据：硬件随时来，在中断/DMA 回调里，是"突发的"
    - 处理数据：任务循环里，是"慢悠悠挨个字节解析的"

  如果没有缓冲，收得快、处理得慢，新数据就把旧数据冲掉了。
  RingBuffer 就是个"环形的队列"，生产端写、消费端读，互不干扰：
    写指针(ulWriteIdx)：生产端往后挪，满了就回到开头（环形）
    读指针(ulReadIdx) ：消费端往后挪，追到写指针就是"空了"

  本工程：DMA 把字节搬进 pucRxbuffer，回调里 uart_fifo_add() 把它们
  逐个写进 RingBuffer；任务里 uc_uart_read_ucbyte() 再逐个读出来喂协议。
  这就是"生产者-消费者"模式，单片机里最常见。

一、你要完成的"通信协议"到底指什么
================================================================================

  本工程的"通信协议" = ANO 协议（自定义协议，帧格式见 ano_ture.c）。
  它跑在 USART1(连飞控) 和 USART3 上。你的任务是理解/维护这条链：

    飞控  ──USART1(DMA)──►  本机 UART 层  ──►  ANO 协议层  ──►  业务(mission等)

  你不需要碰：路径规划算法、屏幕 UI、任务状态机（那是别人/以后的活）。
  你需要吃透：UART vtable 链 + ANO vtable 链 + 两者咬合的两个函数。

三、推荐阅读顺序（新手版，比上一份文件更慢更细）
================================================================================

  第 0 步：先别碰代码，读本文件"零、建立心理模型"两节，吃透中断/DMA/RingBuffer。

  第 1 步：读数据结构（只读 .h）
    types.h        → 看 Point_t 等（协议不太用，跳过也行）
    ring_buffer.h  → 看 stRingBufTdf：读写指针、缓冲指针。
                     懂了"环形队列"就懂了这个结构。
    uart_base.h    → 看 stUartOps（一张函数指针表）+ stUartBase{ops}。
                     这就是"基类"。
    uart_true.h    → 看 stUartRing：基类 + HAL句柄 + RingBuffer + TX队列 + TX buffer。
                     这就是"子类"。注意 stBase 必须是第一个成员（container_of 要求）。
    ano_protocol.h → 看 _dt_st：这是 ANO 协议的"运行时状态"（wait_ck、cmd_send等）。
                     先混个脸熟，不用全懂。
    ano_base.h     → 看 stAnoOps（又一函数指针表）。
    ano_ture.h     → 看 stAnoDevice：基类 + _dt_st指针 + 收发buffer + unique表 + 状态机变量。

  第 2 步：读"装配"（board_init.c，看实例怎么搭起来）
    boart_uart_init.c → 三个 stUartRing 怎么 init、ops 指向哪张表。
                        读完你就知道"USART1 是 DMA 收发"这种结论哪来的。
    ano_board_init.c  → 两个 stAnoDevice 怎么 init、unique 绑哪套实现。

  第 3 步：读"数据流入口"（顺着 main 的调用）
    Ano_Scheduler.c   → driver_init_all() + 循环里调 DrvUartDataCheck()
    Drv_Uart.c         → DrvUartDataCheck()：调 vano_check_data() 喂状态机
                         + 三个 HAL 回调（收/发完成/错误）在这

  第 4 步：读 ANO 协议核心（最重要）
    ano_base.c         → 一堆 vano_*，每个就两行（assert + 转发）。
                         看一遍知道"外部怎么调协议层"。
    ano_ture.c         → 重点！分两块读：
      (a) 收：Data_receive_prepare_Ano_s（7级状态机，逐字节收帧）
              Data_Check_Ano_s（while 循环从 UART 读字节喂状态机）
      (b) 发：vFrame_Send_Ano（拼帧+校验+发）
              Check_to_send_Ano_s（按频率/标志决定发不发）
              CK_Back_Check_Ano_s（ACK 没回就重传，最多5次）
    ano_device_ground.c → 看"收到整帧后怎么处理""发送时怎么填数据"。
                           这是 USART1 的具体业务，最贴近你日常要改的。

  第 5 步：读 UART 实现（配合第4步的咬合点）
    uart_true.c        → uart_ring_read（从 RingBuffer 读一个字节）
                         uart_Ring_tx_DMA_Queue（入队+启动DMA发）
                         uart_Ring_tx_complete_isr_DMA（发完中断里出队续发）
    uart_base.c        → 分发函数，每个两行，快速过。

  第 6 步（按需）：ring_buffer.c 的实现细节、uart_log.c 日志层。


================================================================================
四、ANO 协议帧格式（背下来，看状态机就不晕了）
================================================================================

  每一帧长这样（小端）：

    位置  内容
    [0]   0xAA          帧头1（固定）
    [1]   0xFF          帧头2/目的地址（0xFF=广播）
    [2]   CMD           命令字（如 0xE0命令 0x00确认 0x16航点 0xE2参数...）
    [3]   LEN           载荷字节数（从[4]开始算）
    [4..] 载荷数据      LEN 个字节
    后2   SUM1, SUM2    校验和（见下）

  校验和算法（ano_ture.c:190-196）：
      sum1 = 所有字节累加 & 0xFF
      sum2 = sum1 的累加和 & 0xFF   （每加一个字节，sum2 += 当前的sum1）
    这是 ANO 特有的"两级累加和"，比普通累加和更能检出乱序。

  7 级状态机（ano_ture.c:13-66，逐字节走）：
      state0 收到0xAA → state1
      state1 收到0xFF → state2（否则复位）
      state2 收 CMD   → state3
      state3 收 LEN   → state4（记下要收多少字节载荷）
      state4 收载荷   → 收满 LEN 个 → state5
      state5 收 SUM1   → state6
      state6 收 SUM2   → 复位，调用 Ano_Receive_Anl() 处理整帧
    任何一步对不上 → 复位回 state0（抗噪）。

  发送（vFrame_Send_Ano, ano_ture.c:173-209）就是反过来把上面拼出来。

五、几个单片机新手最容易卡住的点（配代码位置）
================================================================================

【1. container_of 是什么？为什么到处都是？】
  (uart_true.c:10 宏定义)
  vtable 函数收到的参数都是基类指针 stUartBase*，但实现里要访问子类的
  私有字段（如 pUartHandle）。container_of 就是"从基类指针算出子类指针"：
      子类指针 = (基类指针) - (基类在子类里的偏移)
  前提：子类第一个成员必须是基类（uart_true.h:20 的 stBase）。
  理解成 C++ 的 this 还原就行，不用纠结宏的位运算。

【2. "任务级"和"ISR 级"为什么总在强调？】
  - 任务级：在 FreeRTOS 任务里跑，可以阻塞、可以用普通 API。
  - ISR 级：在中断回调里跑，不能阻塞、必须用 FromISR 版本的 API。
  本工程：
    uart_Ring_tx_DMA_Queue   = 任务级发（xQueueSend）
    uart_Ring_tx_complete_isr_DMA = ISR 级发下一帧（xQueueReceiveFromISR）
  两者共用同一个 TX 队列和 TxInProgress 标志，所以 TxInProgress 是 volatile。

【3. TX 队列为什么"非阻塞"？】
  (uart_true.c:86-111)
  调 uart_transmit() 发一帧时，函数立刻返回，不等硬件发完。
  帧被拷进队列，如果当前没在发，就启动一次 DMA；发完中断里再从队列取下一帧。
  好处：上层（ANO）不用等串口慢吞吞发完，能继续干别的。
  代价：队列满了会丢帧（这里队列容量 32，够用）。

【4. wait_ck / ACK 重传是干嘛？】
  (ano_ture.c:266-293)
  发命令帧(0xE0)时，怕对方没收到，发完置 wait_ck=1，等对方回 0x00 确认。
  每 50 个 tick 没收到确认就重发，最多 5 次，超了就放弃（清 wait_ck）。
  这是协议层的"可靠传输"，和 TCP 的 ACK 一个道理。

【5. 为什么有些帧回 ACK 有些不回？】
  - 命令帧(0xE0)、参数帧(0xE2)：要回 ACK（要确认对方执行了）
  - 数据帧(0x16航点等)：不回 ACK（fire-and-forget，量大、丢一帧无所谓）
  看 ano_device_ground.c 里的 switch 就能区分。

【6. HAL 回调怎么和 vtable 连起来？】
  (Drv_Uart.c:57-92)
  HAL 库规定收到数据会调 HAL_UARTEx_RxEventCallback(huart, Size)。
  本工程在这函数里用 if 判断是哪个串口，再调对应 vtable 的 fifo_add。
  发送完成同理（HAL_UART_TxCpltCallback → uart_Ring_tx_complete_isr）。
  这是"HAL 回调 → vtable 分发"的桥，看 Drv_Uart.c 就懂。

六、改代码时该动哪里（协议相关）
================================================================================

  要改的                          →  动哪个文件
  ─────────────────────────────── ─────────────────────────────
  增删一种数据帧(收到/发送)       →  ano_device_ground.c
  改 ANO 帧格式/状态机            →  ano_ture.c (一般不用动)
  换串口/DMA改IT                  →  boart_uart_init.c (改 ucRxIsIT/ucTxIsDMA)
  加一条新串口                    →  boart_uart_init.c + Drv_Uart.c 加 if 分支
  调发送频率                      →  ano_device_ground.c 的 vGround_init_Ano
  调 ACK 重传次数/超时            →  ano_ture.c 的 CK_Back_Check_Ano_s

  绝大多数日常改动都在 ano_device_ground.c：它的 switch(CMD) 管收、
  switch(ucFrame_num) 管发。其他文件基本不用动。


================================================================================
七、一个最小闭环：从飞控发一帧到本机处理完
================================================================================

  1. 飞控从 USART6 发来一帧 0xAA 0xFF 0x01 <len> <数据> <sum1> <sum2>
  2. 本机 USART1 的 DMA 把字节搬进 Ground_RxBuffer
  3. 收到空闲间隔 → HAL_UARTEx_RxEventCallback(uart1, Size)
  4. → uart_fifo_add(ground_uart, Size)
       → uart_ring_add(): 把这几个字节逐个写进 RingBuffer (uart_true.c:121)
  5. 任务循环 → DrvUartDataCheck() → vano_check_data(ground_ano, ground_uart)
  6. → Data_Check_Ano_s: while 从 RingBuffer 读字节，喂给状态机
       (ano_ture.c:327-337)
  7. 状态机收满一帧 → Ano_Receive_Anl() (unique表) → vGround_DT_Data_Receive_Anl_Ano
  8. → switch(CMD) → 对应的 gs_xxx_set() 把载荷存进静态变量 (ano_device_ground.c)
  9. 别的任务通过 gs_xxx_copy() 读走数据

  发送方向反过来：业务调 ground_send_xxx() → 置 WTS → 任务循环 Check_to_send
  → vFrame_Send_Ano 拼帧 → Send_Buff → uart_transmit → 入TX队列 → DMA发
  → 发完中断出队续发。
